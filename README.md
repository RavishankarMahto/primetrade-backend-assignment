## вң… Core Features

### рҹ”җ Authentication & Authorization
- **User Registration** вҖ” email + password with bcrypt hashing (cost factor 12)
- **JWT Login** вҖ” returns signed token, configurable expiry
- **Role-Based Access Control** вҖ” `USER` and `ADMIN` roles with middleware guards
- **Protected Routes** вҖ” JWT required on all `/tasks` and `/admin` endpoints
- **Admin Privileges** вҖ” admins can view all tasks and list all users; users see only their own

### рҹ“Ӣ Task Management (CRUD)
- Create, read, update, delete tasks
- Users access only their own tasks; admins access all
- Tasks linked to owning user via foreign key
- Full validation on every request (title, status, etc.)

### рҹҢҗ API Design
- **Versioned** at `/api/v1` for backward-compatible future evolution
- **Consistent response envelope**: `{ success: true, data: ... }` or `{ success: false, error: { code, message } }`
- Appropriate HTTP status codes on all responses
- **Swagger UI** at `/docs` with full schema documentation
- **OpenAPI JSON** at `/openapi.json` for tooling/import

### рҹ–ҘпёҸ Frontend
- Register & login forms with error/success feedback
- JWT stored in `localStorage`, attached to every API request
- Protected dashboard вҖ” redirects unauthenticated users to login
- Full task CRUD UI (create, view, edit, delete)
- **Admin panel**: "List Users" visible only when role is `ADMIN`
- Error/success messages surfaced from API responses

### рҹ”’ Security & Scalability
- Helmet for HTTP header hardening
- Rate limiting on `/api/*` to prevent brute-force
- Input sanitization & trimming via Zod
- JSON body size capped
- Scalable project structure вҖ” new modules drop into `src/routes/v1` + `src/services`

## вҡЎ Quick Start

**Option A вҖ” Local PostgreSQL:**

Create a database named `primetrade_assignment` in your local PostgreSQL instance.

**Option B вҖ” Docker (recommended for quick start):**

```bash
docker run --name pg-assign \
  -e POSTGRES_PASSWORD=postgres \
  -e POSTGRES_USER=postgres \
  -e POSTGRES_DB=primetrade_assignment \
  -p 5432:5432 \
```

### 2. Backend

```bash
cd backend
```

```env
# Required
```

## рҹ“– API Reference (v1)

Base URL: `http://localhost:4000/api/v1`

### Auth

| Method | Endpoint | Auth Required | Description |
|--------|----------|:---:|-------------|
| `POST` | `/auth/register` | вқҢ | Register new user (bcrypt hashed password) |
| `POST` | `/auth/login` | вқҢ | Login, returns signed JWT |
| `GET` | `/auth/me` | вң… JWT | Get current authenticated user |

### Tasks

| Method | Endpoint | Auth Required | Description |
|--------|----------|:---:|-------------|
| `GET` | `/tasks` | вң… JWT | List tasks (USER: own only В· ADMIN: all) |
| `POST` | `/tasks` | вң… JWT | Create a new task |
| `GET` | `/tasks/:id` | вң… JWT | Get single task (ADMIN can access any) |
| `PATCH` | `/tasks/:id` | вң… JWT | Update task (ADMIN can update any) |
| `DELETE` | `/tasks/:id` | вң… JWT | Delete task (ADMIN can delete any) |

### Admin

| Method | Endpoint | Auth Required | Description |
|--------|----------|:---:|-------------|
| `GET` | `/admin/users` | вң… JWT + `ADMIN` | List all registered users |

### Response Format

**Success:**
```json
{
  "success": true,
  "data": { ... }
}
```

**Error:**
```json
{
  "success": false,
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Invalid or expired token"
  }
}
```

## рҹ”’ Security

| Concern | Implementation |
|---------|---------------|
| **Password Storage** | bcrypt with cost factor **12** |
| **Token Signing** | JWT signed with `JWT_SECRET` (min 32 chars), configurable expiry |
| **Input Validation** | Zod schemas on every endpoint; strings trimmed, types enforced |
| **HTTP Hardening** | Helmet sets secure headers (CSP, HSTS, X-Frame-Options, etc.) |
| **Rate Limiting** | Express rate-limit on `/api/*` вҖ” prevents brute-force attacks |
| **Body Size** | JSON body size capped to prevent payload attacks |
| **RBAC** | Middleware-level role checks вҖ” users cannot escalate privileges |

## рҹ“Ҳ Scalability & Operations

See [`SCALABILITY.md`](./SCALABILITY.md) for a detailed write-up. Summary:

- **Horizontal Scaling** вҖ” Stateless JWT means any backend instance can verify tokens; run multiple instances behind a load balancer (nginx / AWS ALB)
- **Caching** вҖ” Redis layer for frequently-read data (e.g., user profile, task lists) вҖ” reduces DB load
